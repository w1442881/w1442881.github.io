In Your Own Words

1. In your own words, describe the difference between type and instance parameters.

If the object properties (parameters) being modified are of kind instance, changes made to that object will only affect that selected object, i.e., any other objects derived from the same family in the workspace will not be affected. If a family's properties are defined as type properties, by contrast, changes made to one instance’s properties will be reflected in every other instance of that family in the workspace.

a. When do you want to use one type over the other?

It would make sense to set parameters as instance if the property can vary often from one element to the next. Wall lengths and hall widths are examples of instance parameters. If, on the other hand, we’re working with manufactured components whose count we want to track, for example, we’d probably want to differentiate between kinds of components and would therefore not manipulate their dimensions. Light fixtures and screws are examples of families we would probably not modify, or if we do, we’d want all the instances to change.

2. Where do you find families that are loaded into your Revit File?

In the Project Browser, expand Families. Check the Generic category if the family is not based on a modified loaded family.

3. Where do you go to add families that have not been loaded?

Select the Insert menu in the Ribbon. Select Load family in the Load from library panel.

4. How do you place an item from your family library into your Revit model?

Select the view in the Project Browser where you want the item. Find the Family in the Project Browser → Generic. Drag and drop that family from the Project Browser onto the view. A "+" sign will appear under the mouse, meaning that items are ready to be placed when clicked.

5. How do you tell if an item’s dimensions are instance-based or type-based?

In the Properties window while the object is selected, the Type properties will show a description in parentheses. It will typically say “default property”, or, if modified, the value it was modified to. Also, instance properties do not have a Type properties button next to it.

Since Revit is “object-oriented”, we create family object instances in the Revit model the same way we may write a program using an object-oriented programming (OOP) language. To take the analogy one step further, here are some conceptual similarities and differences between a computer language and Revit’s spec: OOP Concept Revit analogy How to access Class definition (reference type) Family types in a model, project (model) types, also title blocks, masses, and annotations† Create a new project or family type Library Families Package a family and distribute the package Class instance Model instance or family instance in a project Create a new project in Revit, drag a family template onto the view Class static state, accessed via property‡ Create a type property for a family and use it in a model All instances’ property value for that family will be affected Class public property (standard) - Instance Create an instance property for a family Only the currently selected instance(s)’ property value will be affected †¬¬¬. Some of these are “singletons” such as the title blocks and occur once in the project. ‡. This is probably implemented in a message queue to affect a private variable in each instance. Most programming languages don’t provide access to classes’ internal state directly. This breaks OOP. One way to create an instance is to drag a specific item from the “Families” section in the Project Browser onto a view of our model. A family template may itself be a model, either created by us or available as a library. These families contain parameters which are classified as being of two kinds: instance or type. Dragging a family onto the Revit workspace creates a new instance and its parameter properties can be modified.
